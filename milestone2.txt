Page 1 MSML/MSAI 605 Milestone 2 (Detailed) What Milestone 2 is testing: Milestone 2 is where your Milestone 1 face verification prototype becomes an iterative machine learning system. You are still working on the same core task from the project plan given two face images, your system should produce a similarity score and a same-person versus different-person decision. In Milestone 1, you built the deterministic plumbing, namely LFW ingestion, deterministic pair generation, and vectorized similarity scoring. In Milestone 2, you keep that reproducible backbone and build the disciplined evaluation loop around it. This means the milestone is not asking for random experimentation. It is asking you to show that you can evaluate the same verification system repeatedly, track those runs, choose an operating threshold deliberately, inspect failure cases, make at least one data-centric improvement, and add validation plus tests so the pipeline behaves reliably from a clean clone. • Experiment tracking, meaning each important evaluation run is recorded clearly enough that a grader can see what changed and what result it produced. • Threshold calibration, meaning you choose the decision threshold using a stated rule on a non-test split, not by guessing or by tuning directly on the final test set. • Error analysis, meaning you look beyond a single accuracy number and study where the verifier fails on same-identity and different-identity pairs. • Data-centric iteration, meaning at least one meaningful improvement is made to the data, pair policy, or data-quality process, not only to code formatting or documentation. • Pipeline reliability, meaning the system performs validation checks, includes tests, and fails early when inputs or outputs are malformed. What you must submit in Canvas • Your Git repository link. • Your Milestone 2 Git tag (example: v0.2). • The link or path in the repository to the Milestone 2 report, if that report location is not obvious from the README. How Milestone 2 fits into this face verification project Students often lose marks in Milestone 2 because they treat it like a generic ML report instead of an extension of the exact system described in the course project. To stay aligned with the project, keep the following picture in mind: • Input to the system: a deterministic set of face pairs, where each pair contains two image paths and a ground-truth label indicating same identity or different identity. • Core scoring output: for each pair, your pipeline produces a similarity or distance value from the representation and scoring logic you built up to this point. • Decision layer: a threshold converts that continuous score into a binary verification decision. • Evaluation layer: the pipeline compares those decisions with ground truth and produces metrics, plots, confusion matrices, and error analyses. • Iteration loop: you compare tracked runs, justify one selected threshold, apply at least one data-centric improvement, then reevaluate under the same reproducible setup. So, in Milestone 2, the central question is not “Can I get one good number?” The question is “Can I operate this verifier like a small, reproducible ML system, evaluate it repeatedly, explain my choices, and improve it in a traceable way?” 2. Important Milestone 2 terms, in project context Term What it means in this milestone Tracked run One recorded evaluation attempt of your face verification pipeline, including what code/config/data version was used and what metrics/artifacts were produced. Threshold sweep Evaluating the same underlying score outputs across a range of threshold values so you can study the tradeoff between false accepts and false rejects. Selected threshold The single operating threshold you finally choose using a stated rule, usually on validation data, before reporting final held-out performance. ROC-style plot A threshold-based tradeoff plot for the verifier, typically showing how true positive behavior changes as false positive behavior changes across thresholds. Confusion matrix The counts of true positives, false positives, true negatives, and false negatives at your selected threshold for a specific split. Error slice A defined subset of the pairs where you inspect failure behavior separately, for example low-quality pairs, visually similar non-matches,

Page 4 5. Step-by-step: Milestone 2 tasks Step 5.1 Lock your Milestone 1 baseline Before you compare runs, decide what the baseline system is. In this project, the baseline should be a concrete, reproducible face verification pipeline that starts from deterministic pairs and produces verification scores in a repeatable way. If you change multiple things at once, you will not know which change caused the difference in results. • Reuse the deterministic split and pair-generation policy from Milestone 1, or regenerate them deterministically using the same documented policy. • Write down which split is used for threshold selection and which split is used for final reporting. A strong default is validation for threshold choice, then test for final reporting. • State clearly what score direction your verifier uses. For example, make it explicit whether larger values indicate more likely same-person pairs or the reverse. • Document the baseline configuration so it can be distinguished later from improved runs. Step 5.2 Decide the threshold-selection rule before final testing The selected threshold is one of the most important decisions in Milestone 2. Do not choose it informally after looking at final test results. Instead, decide in advance how the threshold will be chosen on a non-test split. • Choose a rule such as maximizing balanced accuracy, maximizing F1, minimizing a chosen error tradeoff, or meeting a target false-positive constraint. • State that rule in plain language in your notes, README, and report. • Use the same rule consistently for baseline and improved runs so comparisons stay fair. Step 5.3 Set up tracked runs Milestone 2 requires evidence for at least 5 tracked runs. The tracking method can be simple or sophisticated. What matters is that the record is readable, reproducible, and tied back to the actual face verification pipeline. Acceptable tracking formats include saved run folders inside the repository, a CSV or JSON summary, exported artifacts from a tool such as MLflow or Weights & Biases, or screenshots plus exported artifacts from a tracking interface. Each tracked run should record, at minimum, the following information: Required run field Why it matters Run identifier Lets you refer to one evaluation attempt unambiguously in the report and README. Timestamp Shows run order and supports traceability. Commit hash or code version Connects the result to the exact code that produced it.

Step 5.4 Run threshold sweeps on the validation split A threshold sweep is the cleanest way to study the tradeoff behavior of a verifier. Hold the underlying pair set and score outputs fixed, then vary only the decision threshold. This isolates the effect of the threshold itself. • Use the same pair set and the same score outputs while sweeping thresholds. • Log the threshold values you tried and the metrics produced at each threshold. • Generate the ROC-style plot from this sweep data. • Use the sweep results to understand where false accepts and false rejects start to move in undesirable ways. Step 5.5 Choose and lock the selected threshold After the validation sweep, choose one threshold using the rule you defined earlier. This becomes the operating threshold for the corresponding system version. • Record the chosen threshold in the tracked artifacts, README, and report. • Generate the confusion matrix at that threshold. • Once selected, do not keep retuning it on the final test split. Step 5.6 Make at least one data-centric improvement Milestone 2 explicitly requires at least one data-centric change. In this face verification project, a data-centric change means you improve the quality, composition, filtering, balance, or construction of the evaluation data and pairs. It does not mean only changing variable names, reorganizing files, or restyling the report. Examples of changes that usually count: • Adjusting pair construction so the evaluation set is less skewed or less dominated by a small number of identities. • Filtering unusable, duplicate, or invalid image references in a deterministic way. • Adding deterministic quality checks that remove corrupted or obviously problematic pairs. • Changing the pair policy to better separate training, validation, and test behavior while keeping the process reproducible.

Page 6 • Capping overrepresented identities or adding a more controlled positive-versus-negative sampling policy. Examples of changes that usually do not count by themselves: • Renaming scripts or folders. • Adding comments to code without changing the data or the evaluation setup. • Changing the plot style but not the underlying data or metrics. • Running more thresholds on the exact same data and calling that a data-centric change. Step 5.7 Reevaluate after the data-centric change and compare fairly After introducing the data-centric change, rerun the evaluation using the same tracking discipline. The comparison should be interpretable: the grader should be able to see what changed, which runs belong to the baseline, and which runs belong to the improved version. • Repeat the threshold sweep for the changed data or pair version. • Choose the threshold using the same selection rule you used before. • Compare baseline versus improved results on the appropriate split(s), and mention both gains and tradeoffs honestly. Step 5.8 Perform error analysis with at least two error slices The report must include at least two error slices. An error slice is not just “the model made mistakes.” It is a defined subset of pairs where you inspect the mistakes separately and explain what may be happening. For each slice, include the following: • A clear definition of the slice. • The number of examples in the slice, or at least enough information to show it is a real subset and not one cherry-picked example. • Representative examples from that slice. • A hypothesis explaining why the system fails there. • A short note about what future improvement might help. Possible slice ideas for this project include: • False positives among visually similar different-identity pairs. • False negatives among same-identity pairs with noticeable pose, lighting, or expression variation. • Pairs involving identities with very few images. • Pairs near the decision boundary where scores are ambiguous. • Pairs flagged by a quality proxy or validation rule.

Page 7 Step 5.9 Add validation checks to the pipeline Pipeline validation means your system should check whether inputs, intermediate artifacts, and outputs are valid before continuing. This is part of turning the prototype into a reliable system. Good validation checks for this milestone include: • Pair files follow the expected schema and required columns are present. • Labels are valid binary values and split names are valid. • Referenced image paths exist, or the pipeline fails clearly when they do not. • The number of scores matches the number of evaluated pairs. • Threshold values are in an allowed numeric range. • No accidental split leakage occurs between validation and test artifacts. • Logged metrics are not missing and their denominators are valid. Step 5.10 Add tests: unit tests and one small integration test Milestone 2 requires both unit tests and a small integration test. The key word is small. The grader should be able to run them without needing a huge, slow pipeline. Good unit-test targets include: • Metric calculations from known toy inputs. • Threshold application logic. • Validation logic for pair files or config values. • Confusion-matrix or slice-building utilities. A good small integration test usually does the following: • Uses a tiny deterministic subset or synthetic test fixture. • Runs through the main evaluation path from pairs to metrics and logged artifacts. • Confirms that the pipeline finishes successfully and produces the expected output structure. Do not make the integration test depend on downloading the full dataset during grading unless the README explicitly supports that workflow and it remains practical. Step 5.11 Prepare the 2-page evaluation report The report is short, so it must be selective. The grader should be able to understand what system version was evaluated, how the threshold was chosen, what changed in the data-centric iteration, and what the important failure modes are. A strong 2-page report usually includes: • A brief opening paragraph reminding the reader that this is the Milestone 2 evaluation of your face verification pipeline. • A short description of the baseline setup and the tracked-run process. • The ROC-style plot from the threshold sweep. • The selected-threshold confusion matrix.

Step 5.12 README expectations and clean-clone reproducibility Your README is part of the submission. It must make it easy for a grader to understand the milestone and reproduce the reported result. The README should contain: • A brief project overview describing what the Milestone 2 system does. • A short summary of the baseline and the data-centric improvement. • A How to run section with copy-pastable commands for environment setup, pair generation or loading, evaluation, run logging, and tests. • The location of the report and other important artifacts. • Any notes needed to reproduce the selected threshold and main reported result. Before tagging the milestone, run a clean-clone test. Start from a fresh clone, follow your own README exactly, and confirm that the main outputs and reported result can be reproduced. One acceptable way to reach the required 5 tracked runs There is more than one valid run plan, but students often ask what “5 tracked runs” can look like in practice. The following is one acceptable example. You do not have to use these exact names, but your structure should be similarly traceable. Run Purpose Run 1 Baseline threshold sweep on the validation split. Run 2 Baseline selected-threshold evaluation with confusion matrix recorded. Run 3 Baseline final reporting run on the held-out test split at the locked threshold. Run 4 Post-change threshold sweep after one data-centric improvement. Run 5 Post-change selected-threshold or final reporting run showing the effect of the improvement.

Common mistakes to avoid • Choosing the operating threshold by looking at final test results instead of deciding the rule on a validation split first. • Claiming “5 runs” when the repository only contains one final output and no real tracked history. • Changing several major things at once so the comparison becomes impossible to interpret. • Calling a code cleanup a data-centric improvement even though the data or pair policy never changed. • Including error examples with no slice definition, no counts, or no hypothesis. • Building tests that are too large, too slow, or dependent on manual local setup. • Leaving the report path unclear, forgetting to mention artifacts in the README, or tagging the wrong commit.
